# Contact Page UX Overhaul Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Demote the contact form behind the booker, trim form friction, and fix mobile crowding before next-week launch — turning the contact page into one obvious decision (book a slot) with a quiet escape hatch (write a message instead).

**Architecture:** Pure HTML/CSS edits in `contact.html`. No backend changes — the Express `/api/contact` handler already null-safe-handles missing form fields. Cache-buster bump across all 8 HTML pages per `docs/deploy.md` discipline. Doc sync in `docs/pages.md`.

**Tech Stack:** Static HTML, Tailwind CSS v4, Express (untouched).

**Scope source:** `docs/superpowers/specs/2026-05-18-contact-booking-widget-design.md` (the v1 spec just shipped) + the UX audit from 2026-05-18 (in-conversation, not yet persisted).

**Out of scope (per the audit's "Defer until after launch"):**
- WTSAdmin mirror of bookings via Calendar-API poller (still in `_backlog.md`).
- Branded booker (Cal.com / SavvyCal). Google's UI is good enough until measured otherwise.
- Wiring Plausible/PostHog analytics (this plan adds the `data-cta` markers so the future wiring is markup-free).
- Pre-call intake automation; reminders beyond Google's defaults.
- CSP `frame-src` (no CSP in repo today).

**Branch:** continue on `feat/contact-booking-widget` (the booker work is on this branch; this overhaul ships as additional commits on the same branch).
